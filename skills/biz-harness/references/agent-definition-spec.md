# Agent Definition Specification

Mandatory format for all agent files generated by BizHarness. Every generated
agent MUST follow this specification to work with Claude Code's agent system.

## File Location

```
.claude/agents/{agent-name}.md
```

Agent names: lowercase, hyphenated, descriptive of role (not technology).
- GOOD: `ticket-classifier.md`, `content-generator.md`
- BAD: `agent1.md`, `llm-processor.md`, `the-classifier-agent.md`

## Required YAML Frontmatter

```yaml
---
name: {agent-name}
model: {opus | sonnet | haiku}
description: >-
  {Pushy description — see skill-writing-guide.md for how to write these.
  Must cover all scenarios where this agent should be invoked.}
tools:
  - {Tool1}
  - {Tool2}
---
```

### Model Selection Guide

| Model | When to Use | Cost | Quality |
|-------|-------------|------|---------|
| opus | Complex reasoning, synthesis, content generation, final decisions | High | Highest |
| sonnet | Classification, search, routing, validation, orchestration | Medium | High |
| haiku | Simple transformations, formatting, data extraction | Low | Good |

**Default rule:** Use `sonnet` unless the task requires complex reasoning
(use `opus`) or is purely mechanical (use `haiku`).

### Tool Access

Only grant tools the agent actually needs. Principle of least privilege.

| Tool | Grant When |
|------|-----------|
| Read | Agent needs to read files |
| Grep | Agent needs to search file contents |
| Glob | Agent needs to find files by pattern |
| Write | Agent needs to create files |
| Edit | Agent needs to modify existing files |
| Bash | Agent needs to run shell commands |
| Agent | Agent needs to dispatch sub-agents (orchestrators only) |
| WebSearch | Agent needs to search the internet |
| WebFetch | Agent needs to fetch specific URLs |

**Never grant:** All tools, or tools "just in case." Each unnecessary tool
expands the agent's action space and increases risk of unintended behavior.

## Required Sections in Agent Body

### 1. Header Comment (auto-generated)

```markdown
<!-- Generated by BizHarness v{version} -->
<!-- Use Case: {UC-ID} {use case name} -->
<!-- Role: {role description} ({pattern name}) -->
```

### 2. Role Title and Purpose

```markdown
# {Role Name}

You {one sentence describing what this agent does and why}.
```

### 3. Why This Role Exists

```markdown
## Why This Role Exists

{2-3 sentences explaining the business reason for this agent's existence.
What goes wrong if this role is missing? What value does it provide?}
```

**Why this section is required:** Agents that understand their purpose make
better judgment calls in edge cases. Without it, the agent follows rules
mechanically and breaks on novel inputs.

### 4. Input Contract

```markdown
## Input Contract

You receive:
- `{field_name}`: {description and format}
- `{field_name}`: {description and format}
```

Define exactly what data the agent receives, in what format, from whom.

### 5. Output Contract

```markdown
## Output Contract

Return {format} in exactly this structure:

​```yaml
{concrete output schema with example values}
​```
```

**Rules:**
- Use YAML for structured output (easier for LLMs to generate consistently)
- Include example values, not just field names
- Specify constraints (max length, valid values, ranges)

### 6. Business Rules / Decision Logic

```markdown
## {Rules Section — name varies by role}

{The specific rules, criteria, or logic this agent applies.
Tables for lookup data. Decision trees for branching logic.
Always include WHY for each rule.}
```

### 7. Constraints

```markdown
## Constraints

- {Hard constraint with reasoning}
- {Hard constraint with reasoning}
```

Constraints are things the agent must NEVER do, regardless of input.

## Data Passing Between Agents

### Protocol Options

Choose the right data passing method based on your architecture pattern:

#### 1. Task Description Passing (Default)

Pass data via the prompt/description when dispatching an agent.
Best for: Pipeline and Fan-out patterns with moderate data sizes.

```markdown
# In orchestrator:
Dispatch to Agent B with:
- classification: {paste Agent A's output}
- original_input: {paste original input}
```

#### 2. File-Based Passing

Write intermediate results to `_workspace/` directory.
Best for: Large data, multiple consumers, or when audit trail is needed.

```markdown
# Agent A writes:
Write output to _workspace/{uc-id}/stage-1-output.yaml

# Agent B reads:
Read _workspace/{uc-id}/stage-1-output.yaml
```

**Workspace convention:** `_workspace/{use-case-id}/` for intermediate artifacts.
Clean up after final output is produced.

#### 3. Message-Based Passing (Agent Teams only)

Use `SendMessage` for back-and-forth between agents.
Best for: Producer-Reviewer and Supervisor patterns.

```markdown
# Producer sends to Reviewer:
SendMessage(to: "reviewer", content: {draft output})

# Reviewer responds:
SendMessage(to: "producer", content: {feedback})
```

### Protocol Selection Guide

| Pattern | Recommended Protocol | Reason |
|---------|---------------------|--------|
| Pipeline | Task Description | Simple, sequential, moderate data |
| Fan-out/Fan-in | Task Description | Parallel dispatch, results collected by orchestrator |
| Expert Pool | Task Description | Router dispatches to one expert |
| Producer-Reviewer | Message-Based | Back-and-forth revision loop |
| Supervisor | Message-Based | Dynamic task assignment |
| Hierarchical | File-Based | Large data across sub-teams |

## Execution Modes

### Subagent Mode (Default)

Uses the `Agent` tool to dispatch work. Each agent runs as a subprocess.

```markdown
# Orchestrator dispatches:
Agent(prompt: "Classify this ticket: {ticket}", subagent_type: "general-purpose")
```

**When to use:** Most use cases. Simpler setup, no experimental flags needed.

### Agent Teams Mode (Experimental)

Uses `TeamCreate`, `SendMessage`, `TaskCreate` for persistent, collaborative agents.

```markdown
# Create team:
TeamCreate(name: "support-team", agents: ["classifier", "searcher", "generator"])

# Send work:
SendMessage(to: "classifier", content: "Classify: {ticket}")
```

**When to use:** Supervisor and Producer-Reviewer patterns where agents need
ongoing back-and-forth. Requires `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`.

### Mode Selection

| Need | Use |
|------|-----|
| Simple sequential or parallel dispatch | Subagent Mode |
| Back-and-forth conversation between agents | Agent Teams Mode |
| Dynamic task reassignment | Agent Teams Mode |
| One-shot analysis per agent | Subagent Mode |
