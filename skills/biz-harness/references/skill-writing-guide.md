# Skill Writing Guide

How to write effective skills for agents generated by BizHarness. Follows the
why-first principle and progressive disclosure pattern.

## Core Principles

### 1. Why-First Instructions

Every rule in a skill must explain WHY it exists. An LLM that understands the
reasoning can handle edge cases the rule didn't anticipate.

```markdown
# BAD — rule without reasoning
Always validate email format before processing.

# GOOD — why-first
Validate email format before processing. Invalid emails cause silent failures
in the notification pipeline — the message is queued but never delivered,
and the customer sees no error.
```

**Why this matters:** Rules without reasoning become cargo cult. The LLM follows
them mechanically but breaks in novel situations. When it knows WHY, it can
generalize correctly.

### 2. Pushy Descriptions

The `description` field in the YAML frontmatter determines when Claude Code
activates your skill. Write it aggressively — err on the side of triggering too
often rather than too rarely.

```yaml
# BAD — passive, narrow triggers
description: "Handles product listing creation"

# GOOD — pushy, covers many phrasings
description: >-
  Create, edit, validate, or optimize product listings for any marketplace.
  Use when uploading products, writing descriptions, checking listing quality,
  fixing rejected listings, generating SEO tags, or managing catalog data.
  Also use when the user mentions "listing", "catalog", "product page",
  "SKU", or "marketplace submission".
```

**Checklist for descriptions:**
- [ ] Lists the primary actions (verbs the user might use)
- [ ] Lists alternative phrasings and synonyms
- [ ] Lists keywords that should trigger activation
- [ ] Covers both creation and modification scenarios
- [ ] Mentions what to do (not just what the skill is about)

### 3. Progressive Disclosure (3-Tier Context Loading)

Context window is a shared resource. Load only what's needed, when it's needed.

```
Tier 1: Metadata (name + description)     ~100 words  | Always loaded
Tier 2: SKILL.md body                     <500 lines  | Loaded on trigger
Tier 3: references/ files                 Unlimited   | Loaded on demand
```

**Rules:**
- SKILL.md must stay under 400 lines (target 200-300)
- Heavy content (catalogs, examples, lookup tables) goes in `references/`
- Reference files are loaded with explicit instruction: "Load references/X.md"
- Never pre-load all references — load only what the current phase needs

### 4. Separation of Who and How

- **Agent files** define WHO does the work (role, constraints, I/O contract)
- **Skill files** define HOW to do it (domain knowledge, decision criteria)

An agent should reference its skill, not duplicate the knowledge:

```markdown
# Agent file (WHO)
You are a ticket classifier. Use the support-domain skill for category
definitions and escalation policies.

# Skill file (HOW)
## Ticket Categories
| Category | Description | Typical Resolution |
...
```

**Why separate?** Multiple agents can share a skill. When domain knowledge
changes, update the skill once instead of every agent.

## Structure Template

### Skill File Structure

```markdown
---
name: {domain}-{function}
description: >-
  {Pushy description covering all trigger scenarios}
---

# {Skill Name}

{1-2 sentence purpose statement with WHY this knowledge exists}

## {Section 1: Core Domain Knowledge}

{Tables, rules, definitions that agents need to make decisions}

## {Section 2: Decision Criteria}

{When to choose A vs B, thresholds, priority rules}

## {Section 3: Constraints & Policies}

{Hard rules that must never be violated, with reasoning}
```

### What Goes in a Skill vs. What Goes in an Agent

| Content Type | Skill | Agent |
|-------------|-------|-------|
| Domain vocabulary | YES | NO |
| Business rules | YES | NO |
| Decision thresholds | YES | Referenced |
| Output format | NO | YES |
| Tool access | NO | YES |
| Role description | NO | YES |
| I/O contracts | NO | YES |
| Escalation policies | YES | Referenced |

## Anti-Patterns

### 1. Overfitting to Specific Inputs

```markdown
# BAD — overfitted to one product type
If the product is a "Blue Widget Model X", classify as "electronics > widgets".

# GOOD — generalizable rule
Classify products by matching attributes (material, function, target user)
against the category taxonomy. When multiple categories match, prefer the
most specific category that has > 10 existing products.
```

### 2. Instruction Soup

```markdown
# BAD — wall of text with no structure
When processing tickets you should first check if the customer is enterprise
then look at the urgency and also check the category and make sure to
validate the email and don't forget to check for duplicate tickets...

# GOOD — structured, scannable
## Processing Order
1. Check customer tier (enterprise gets priority routing)
2. Classify category and urgency
3. Check for duplicate tickets (same customer, same subject, last 7 days)
4. Validate contact information
```

### 3. Redundant Context

```markdown
# BAD — repeating what the agent definition already says
You are a ticket classifier. Your job is to classify tickets.
You will receive tickets and classify them into categories.

# GOOD — add knowledge the agent doesn't have
## Category Definitions
| Category | Signals | Example Ticket |
| bug | "broken", "error", "crash", "doesn't work" | "Login page returns 500" |
| how_to | "how do I", "where is", "can I" | "How do I export my data?" |
```

### 4. Missing Why

```markdown
# BAD
Maximum 5 tags per product.

# GOOD
Maximum 5 tags per product. Marketplace search indexes only the first 5 tags;
additional tags are ignored and dilute relevance scoring.
```

## Quality Checklist

Before finalizing a skill:

- [ ] Description is pushy (covers synonyms, alternative phrasings, keywords)
- [ ] SKILL.md is under 400 lines
- [ ] Heavy content is in `references/` with on-demand loading
- [ ] Every rule has a WHY explanation
- [ ] No overlap with agent definition files (who vs how separation)
- [ ] Tables are used for lookup data (not prose)
- [ ] Anti-patterns avoided (overfitting, instruction soup, redundancy)
- [ ] Domain terminology is consistent across all skill files
